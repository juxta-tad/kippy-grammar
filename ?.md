No—but you do need **one small hook per effect type**. The key is to make `?` lower to a **single operation name** that types can implement, rather than hard-coding `Maybe`, `Result`, `Parser`, etc.

## The idea

Lower

```kippy
expr?
```

to something like:

```kippy
Context.bind(expr, fn x => CONTINUE(x))
```

or equivalently:

```kippy
and_then(expr, fn x => CONTINUE(x))
```

Then each type only needs to implement **one function** (often called `bind`, `and_then`, or similar).

You do **not** implement `?` separately for every type.

## Example: Result

Define once:

```kippy
ability Bind(m)
	bind: fn(m[a], fn(a) -> m[b]) -> m[b]
```

Implementation:

```kippy
extend Result[a, e] with Bind
	bind ra, f =>
		match ra to
			Err(e) -> Err(e)
			Ok(x) -> f with x
```

Now `?` works automatically for `Result`.

## Example: Maybe

```kippy
extend Maybe[a] with Bind
	bind ma, f =>
		match ma to
			None -> None
			Some(x) -> f with x
```

No new compiler logic needed.

## Example: Parser

```kippy
extend Parser[a] with Bind
	bind pa, f =>
		fn input =>
			match pa with input to
				Fail(e) -> Fail(e)
				Ok(x, next) -> f with x with next
```

Again: same interface.

## Compiler lowering

Code:

```kippy
let
	a = parse_a?
	b = parse_b with a?
in
	Ok(#(a; b))
```

Compiler roughly rewrites to:

```kippy
bind(parse_a, fn a =>
	bind(parse_b with a, fn b =>
		Ok(#(a; b))))
```

So the compiler only knows:

```
?  →  bind
```

Everything else is library code.

## What else you need

Just one more helper: something like `pure` (or equivalent) so the final value can be wrapped back into the context.

Example for `Result`:

```kippy
pure x = Ok(x)
```

But if your final expression already returns the contextual type (`Ok(...)`, `Some(...)`, parser value), you may not even need that.

## Practical answer

You **do not need loads of implementations**.

You need:

1. One **ability/interface** (`Bind` / `Chain` / `Effect`)
2. One **method** (`bind` / `and_then`)
3. One implementation per effect type

That is usually only **3–5 implementations in the whole ecosystem**:

* `Maybe`
* `Result`
* `Parser`
* `Async` / `Task`
* maybe `List`

After that, `?` works everywhere automatically.

## Important design note

This is exactly the minimal mechanism behind:

* Rust `?`
* Haskell `do`
* Swift `try`
* Kotlin `suspend` chains

They all compile to a **single bind-like operation**.

So the short answer is:

**No. The compiler only needs to know how to call `bind`. Each type implements that once, and `?` works automatically.**


The point is that **`build` and `?` solve two different composition problems**.

They are not alternatives. They cover two different shapes of computation.

---

# 1. What `build` solves

`build` solves **independent composition**.

You have multiple computations that **do not depend on each other**, and you want to combine their results into a record.

Example:

```kippy
build Config {
	host <- parse_host with text
	port <- parse_port with text
	ssl  <- parse_ssl with text
}
```

Each field can be computed independently.

Conceptually this is:

```
map3(parse_host, parse_port, parse_ssl)
```

Important property:

**fields cannot depend on each other**

This means `build` works great for:

* constructing records
* validating forms
* parsing fixed structures
* accumulating errors
* parallelizable work

---

# 2. What `?` solves

`?` solves **dependent sequencing**.

You need the result of one step **before deciding the next step**.

Example:

```kippy
let
	len = parse_length?
	body = parse_exactly with len?
in
	Ok({ len = len; body = body })
```

Here:

```
body depends on len
```

This cannot be expressed with `build`.

Because `build` assumes fields are independent.

---

# 3. Why both are needed

They represent two fundamental computation shapes.

### Independent composition

```
A
B
C
```

Combine results.

```
(A, B, C) -> Record
```

That is `build`.

---

### Dependent sequencing

```
A -> produces x
B(x) -> produces y
C(y) -> produces z
```

Each step depends on the previous one.

That is `?` + `let`.

---

# 4. What code looks like in practice

## Using `build`

```kippy
build User {
	name  <- parse_name with form
	email <- parse_email with form
	age   <- parse_age with form
}
```

All fields are independent.

---

## Using `?`

```kippy
let
	n = parse_int?
	items = parse_n_items with n?
in
	Ok(items)
```

The second step depends on the first.

---

# 5. Using them together

Example parser:

First read a tag, then build a structure.

```kippy
let
	tag = parse_tag?
in
	if tag == Person then
		build Person {
			name <- parse_name
			age  <- parse_age
		}
	else
		build Company {
			name <- parse_company_name
			size <- parse_company_size
		}
```

Flow:

1. sequential step (`?`)
2. then independent record assembly (`build`)

---

# 6. Why languages separate these

Because the two compositions have **different mathematical properties**:

| pattern | property                |
| ------- | ----------------------- |
| build   | independent/applicative |
| ?       | dependent/monadic       |

You don't need to expose that terminology to users, but the distinction is real.

---

# 7. The real payoff

Once you have both:

### Code stays flat

Without them you get:

```
match
	match
		match
```

With them you get:

```
let
	x = ...
	y = ...
in
	build ...
```

or

```
build ...
```

depending on the situation.

---

# 8. The short version

`build` = **assemble pieces**

`?` = **step through a sequence**

Both are necessary because programs naturally contain **both patterns**.

---

If you're designing the language, the most important rule is:

> `build` should **not allow dependency between fields**.

That single restriction keeps the language simple and prevents the feature from becoming messy.


1. Destructuring bind must be total, otherwise runtime error

Example:

let [a; b] = xs
in a + b

If xs is not exactly two elements, execution traps.

This is the simplest rule, but it is the weakest design. It makes ordinary destructuring feel unsafe.

2. Destructuring bind is only allowed for patterns that are statically irrefutable

This is the strongest design.

Allow:

let {name; age} = person
in ...

if records always have those fields.

Allow:

let #(x; y) = point
in ...

if tuples always have fixed arity.

Allow:

let [a; b] = xs
in ...

only if list length is known by type, otherwise reject.

Disallow refutable cases like:

let Some(x) = maybe
in x

because Some(x) might fail.

Then require match for refutable patterns:

match maybe to
  Some(x) -> x
  None -> 0

This gives a very clean rule:

let is for guaranteed binding

match is for branching on uncertainty
